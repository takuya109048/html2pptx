# 推奨章立てはAIの探索と判断を助ける
## What、How、Workflow、Gotchasを地図として渡す

| key | value |
|-----|-------|
| layout | table_conclusion |
| note | では、実際に何を書けばAIは迷いにくくなるのか。ここでは章立てを、単なる整理ではなく作業地図として見る。プロジェクト概要は目的地を示し、技術スタックは使う道具を固定する。ディレクトリ構成はファイル探索の手掛かりになり、コマンド集は検証手順へ直結する。さらに作業ルールと落とし穴を入れると、AIは変更前後の判断まで揃えやすくなる。この章立てがあることで、依頼ごとに探すべき情報がぶれにくくなる。 |

```table
| 章 | 役割 | 入れるべき情報 |
| --- | --- | --- |
| **プロジェクト概要** | 何を作るかを理解させる | 目的、ユーザー、重要ユースケース |
| **技術スタック** | 提案の前提を揃える | 言語、FW、UIライブラリ、禁止ライブラリ |
| **ディレクトリ構成** | ファイル探索の地図にする | 主要フォルダの役割、隔離ルール |
| **コマンド集** | 実行計画へ組み込ませる | `dev`、`test`、`lint`、`typecheck` |
| **作業ルール** | AIの行動を制御する | 調査、計画、検証、要約時の保持事項 |
| **落とし穴** | 現場固有の事故を防ぐ | 本番環境、個人情報、シェル差分 |
```
```conclusion
章立ては情報分類ではなく、AIが次に何を読むべきかを迷わないための作業地図である。
```
