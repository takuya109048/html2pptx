## STORY_ANALYSIS — GLOBISストーリー分析手法

### Step 2-1: プレゼンテーション文脈の把握

| 項目 | 確認内容 |
|------|---------|
| 発表目的 | 情報共有 / 説得・提案 / 依頼 / 報告 |
| 対象オーディエンス | 社内（上司/同僚） / 顧客 / 経営層 / 一般 |
| 発表時間 | 5分 / 10分 / 15〜20分 / 30分以上 |

**発表時間→スライド枚数の目安（cover含む）:**

| 発表時間 | 推奨枚数 |
|---------|---------|
| 5分 | 5〜7枚 |
| 10分 | 8〜12枚 |
| 15〜20分 | 12〜18枚 |
| 30分以上 | 18〜25枚 |

### Step 2-2: フレームワーク選択

| 目的 | フレームワーク | 構造 | 適したシーン |
|------|--------------|------|------------|
| 要点を端的に伝える | **SDS法** | Summary → Details → Summary | 短いスピーチ、忙しい上司への報告 |
| 説得・提案 | **PREP法** | Point → Reason → Example → Point | 提案書、説得場面 |
| 依頼・言いにくい内容 | **DESC法** | Describe → Express → Suggest → Choose | 依頼、交渉 |
| 汎用・長尺プレゼン | **序論→本論→結論** | 導入 → 主張+根拠 → まとめ | 一般的なビジネスプレゼン |
| 営業プレゼン | **AIDMA応用** | Attention → Interest → Desire → Memory → Action | 顧客向け提案 |

### Step 2-3: ストーリー骨子設計

**1. メインメッセージの定義**
ソース全体を通じて「聴衆に最も伝えたい1文」を定義する。

**2. フレームワークに沿ったセクション分割**

PREP法の例:
- Point（結論）: AIで開発生産性50%向上
- Reason（理由）: コード生成・レビュー・テストの自動化
- Example（具体例）: 導入事例・数値データ
- Point（結論の再述）: 早期導入が競争優位につながる

**3. 各セクションの内容抽出**

ソースから以下を抽出・整理する:
- 主張・メッセージ（スライドタイトルになる）
- 根拠・理由（本文箇条書きになる）
- データ・事例（tableやカードの内容になる）
- プロセス・手順（flowステップになる）
- 並列要素（3カード系になる）
- 結論・まとめ（conclusionになる）

**4. スライド構成案の内部作成**

2枚目は必ず `plain_2col` の目次にする。目次はスライドタイトルの単純羅列にしない。スライド間でカテゴリ分けできる単位を大見出しにし、その配下に各スライドタイトルを小見出しとして置く。配置は左右均等分割ではなく、左カラムへ上から順に積み、左に入りきらない残りだけ右カラムへオーバーフローさせる。
```
### 導入
- デッキタイトル
- 背景・現状
### 分析
- 詳細・根拠
- 比較・論点
### 結論
- まとめ・次の判断
```

**5. ユーザー確認は行わない**
ターンBでは構成案の確認質問を挟まず、PPTX生成まで進める。

## TEMPLATE_WORKFLOW — テンプレート選択決定木

各スライドに以下の決定木を**上から順番に**適用する。最初に一致した条件でテンプレートを決定し、それ以降のルールは確認しない。どれにも当てはまらない場合のみ `plain_1col` / `plain_2col` を使う。**強引な当てはめは行わない。**

```
1. タイトル/表紙スライドか？
   → cover

2. 時系列・フェーズ変遷など「流れ」があり3段階か？
   → flow_3step

3. 時系列・フェーズ変遷など「流れ」があり4段階か？
   → flow_4step

4. 並列する3つの要素・特徴・メリット・施策がフラットに並ぶか？
   → list_3card

5. 1つの共通テーマ・課題から3つのアプローチ・方向性に展開するか？
   → diffuse_3card

6. 3つの根拠・証拠・理由が1つの結論に収束するか？
   → converge_3card

7. 背景・文脈の説明＋3つのポイントという構成か？
   → bg_3card

8. 表形式のデータがあり、かつ1つの重要な結論・示唆があるか？
   → table_conclusion

9. 表形式のデータのみで結論は別スライドか？
   → table

10. 2つのものを3つの観点で対比・比較するか？
    → compare_2col_3row

11. マトリックス形式のデータか？
    - 3×3グリッドで縦方向にフロー → flow_matrix_3x3
    - 3×3グリッドで横方向にフロー → h_flow_matrix_3x3
    - 3×2グリッドで横方向にフロー → h_flow_matrix_3x2
    - 4×2グリッドで横方向にフロー → h_flow_matrix_4x2
    - 単純な3×3表 → matrix_3x3

12. 2つの対立・比較・対照する概念を並べて説明するか？
    → plain_2col

13. それ以外（単一コンセプトの説明・段落テキスト）
    → plain_1col
```

### nanobanana2使用時の置換ルール（厳守）

nanobanana2使用時、目次以外のplain系は最終DECK_MDに残さない。手順12・13でplain系になりそうなスライドは、以下の基準で置換または再設計する。

| plain候補 | 置換・再設計ルール |
|-----------|----------------|
| `plain_1col` | `plain_image_col` に置換し、概念イラスト用の `image_label_1` を入れる |
| `plain_2col` | `plain_image_col` に置換する。両カラム必須なら `table` / `compare_2col_3row` / `list_3card` へ再設計する |

**最終チェック:**
- Yes時のDECK_MDに目次以外の `plain_1col` / `plain_2col` が残っていたら失敗である。
- 画像が不自然な場合でもplainへ戻さず、`table` / `table_conclusion` / `list_3card` / `flow_*` など画像不要レイアウトへ再設計する。
- 置換先の画像スライドは `plain_image_col` のみとし、必ず `image_label_1` を入れる。

## MD_SYNTAX — MD記法リファレンス

### 基本構造

```markdown
# スライドタイトル
## サブタイトル（メッセージ）

| key | value |
|-----|-------|
| layout | <レイアウト名> |
| note | 発表者ノート |

\```card-a
### カード見出し
- 箇条書き1
- **強調テキスト**
\```

---
```

- スライドは `---` で区切る
- `# タイトル` は必須（`## メッセージ` はオプション）
- フロントマターはMarkdownテーブル形式
- **⚠️ 重要:** card-a等のコンテンツブロック内に `---` を書いてはいけない。パーサーがスライド区切りとして解釈し、直後のスライドが layout を失う。区切り線が必要な場合は空行または `####` 見出しで代替する

## NANOBANANA_RULES — nanobanana2プロンプト生成ルール

### ルールA: image_label への画像プロンプト

**対象テンプレート:** `plain_image_col` のみ
**書き込み先:** フロントマターテーブルの `image_label_1` フィールド

**image_col（6:5 ほぼ正方形）のプロンプト形式:**
```
[スライドテーマ]を表すシンプルなイラスト。フラットデザイン、ミニマルアイコン風、ビジネス向け、白背景。6:5の比率で[具体的な概念]を1枚のビジュアルで表現する。文字なし。
```

### ルールB: note へのアイコン一括生成プロンプト

**対象テンプレート:** `list_3card`, `flow_3step`, `flow_4step`, `diffuse_3card`, `converge_3card`, `bg_3card`
**書き込み先:** `note` フィールドの末尾に ` --- ` で区切って追記（読み上げ原稿と画像生成プロンプトを視覚的に分離するため）

書き込み形式（テーブル1行に収める）:
`読み上げ原稿テキスト。 --- [nanobanana2 icon prompt] プロンプトテキスト`

```
[nanobanana2 icon prompt]
[N]つのフラットアイコンを横一列に並べた1枚の画像を生成する。
左から順に:
  アイコン1: [カードAのテーマを表す概念・シンボル]
  アイコン2: [カードBのテーマを表す概念・シンボル]
  アイコン3: [カードCのテーマを表す概念・シンボル]
各アイコンは正方形の枠内に収まるシンプルなフラットデザイン。白背景、色統一、ミニマルスタイル、文字なし。
※生成後に水平[N]等分にトリミングして各カード/ステップのアイコンとして使用。
```

| テンプレート | アイコン数 |
|------------|----------|
| `list_3card`, `flow_3step`, `diffuse_3card`, `converge_3card`, `bg_3card` | 3 |
| `flow_4step` | 4 |

### プロンプト品質ガイドライン

1. **具体性**: 「ビジネス」ではなく「会議室でのプレゼン」など具体的なシンボル・シーンを指定
2. **スタイル統一**: 全スライドで「フラットデザイン、白背景、ミニマル」を一貫させる
3. **アスペクト比の明示**: `plain_image_col` 用に「6:5の比率」を必ず明記
4. **テキスト不要**: 画像内にテキスト・文字を含めないよう「文字なし」を追記する

## CONTENT_LIMITS — カードコンテンツ密度目標

カードは件数・視覚行数・本文文字量を同時に見る。list_3cardは約17字×10行、diffuse/flow系は約17字×6行、flow_4stepは約11字×6行が限界である。内容が薄い時はレイアウト変更や分割を選ぶ。

| レイアウト | 通常目標 | 本文視覚行 | 文字量 |
|-----------|----------|------------|--------|
| list_3card | リード＋3〜4項目 | 6〜8 | 100字以上 |
| diffuse_3card / bg_3card / flow_3step | リード＋2〜3項目 | 4〜5 | 58字以上 |
| converge_3card | リード＋2項目＋結論 | カード4〜5・結論2以下 | 58字以上 |
| flow_4step | 短いリード＋2項目 | 3〜4 | 40字以上 |
| plain_2col | リード＋3〜4項目/列 | 6〜9 | 115字以上 |
| plain_1col | 段落＋箇条書きで余白なく埋める | — | 180字以上 |

- 箇条書きは原則20〜38字程度。flow_4stepや結論付き3cardでは短めにし、1項目が2行を超えないようにする。
- 3cardで説明が薄くなる場合は、無理に圧縮せずplain_2col/table/分割スライドへ逃がす。
- converge_3cardはカード本文と結論が競合するため、カード内に`####`や長い補足段落を置かない。
- 表や比較マトリクスのセルも単語だけにせず、「何がどう違うか」が分かる句にする。
- nanobanana使用時はアイコン設置スペース（約2行分）を確保する。

## CONTENT_VARIATION — 単調な箇条書きの回避

カード/カラムはMarkdown装飾を多用し、構造化と強調で意味を読みやすくする。

- 各 `card-*` / `step-*` は `###` の直後に1文のリード文を書く。
- 重要語は `**太字**`、補足・定義は *斜体*、コードやファイル名は `コード`、旧情報は ~~打ち消し~~ で示す。
- 箇条書きはスライド内で2種以上混ぜる。例: 短文項目、`A → B`、`従来/今後`、問いかけ、`**語句**を文中で強調`。
- `**観点:** 補足`は多用せず、番号リスト、短い`####`小見出し、対比表現も混ぜる。
- list_3card/plain_2colでは、短い補足文や`####`小見出し、番号リストも使える。
- converge/diffuse/bg/flow系は、リード文＋2項目を基本にし、`####`を原則避ける。

## SLIDE_STYLE — である調

スライドに表示されるタイトル、カード、表、結論はすべてである調にする。`です/ます/ください/しましょう/〜します`を避け、`〜である/〜する/〜となる`へ直す。noteも原則である調に寄せる。

## SPEAKER_NOTES — 読み上げ原稿品質

noteはPPTX発表者ビューに入る読み上げ原稿であり、スライド本文を見ていない人にも意味が通じる補助説明を書く。

### noteの必須要素

- 1枚につき原則180〜320字程度。coverのみ120字以上でよい。
- 冒頭でそのスライドの主張を1文で述べる。
- 本文のカード・表・フローを順番に説明し、各要素の意味や関係を補う。
- 最後に「したがって何を理解すべきか」「次のスライドとどうつながるか」を示す。
- スライド上の短い箇条書きをそのまま読み上げるだけにしない。

### noteの型

```
このスライドで伝えたいことは、[主張]である。まず[要素A]は[意味]を示す。次に[要素B]は[理由/対比]である。最後に[要素C]によって[結論]が分かる。つまり[聴衆が取るべき解釈]である。
```

## DENSITY_REVIEW — プロンプト自己点検手順

B-3ではPythonによる密度検証を行わない。DECK_MDを作った直後に、以下の観点を内的に確認し、問題があればDECK_MD本文を直接直してからPPTX変換へ進む。

### レイアウト別の目安

| レイアウト | 自己点検の目安 |
|-----------|----------------|
| list_3card | 各カードにリード1文＋3項目。カード間の文量差を小さくする |
| diffuse_3card / bg_3card / flow_3step | section/stepの主張を1文で置き、各カードはリード＋2項目に収める |
| converge_3card | 各カードはリード＋2項目まで。結論は2行相当の短文にする |
| flow_4step | 1ステップは短いリード＋1〜2項目。長文説明は避ける |
| plain_2col | 各カラムにリード＋3項目程度を置き、左右の密度を揃える |
| plain_1col | 段落＋箇条書きで余白を埋め、短いメモだけのスライドにしない |

### 修正チェックリスト

- 余白が目立つカードには、判断基準・注意点・影響を1項目追加する。
- はみ出しそうなカードは、例示や修飾語を削り、必要ならplain_2col/tableへ変更する。
- `**ラベル**: 説明`の連続は避け、短文、矢印、対比、問いかけを混ぜる。
- 装飾・番号・小見出しが少ないスライドは、重要語を強調して視線の入口を作る。
- noteは読み上げ原稿だけで意味が通るよう、主張・各要素の意味・結論を含める。
- スライド表示文はである調に統一する。
- nanobanana2使用時はアイコン分の余白を残し、本文を詰め込みすぎない。
- nanobanana2使用時はplain_1col/plain_2colを最終DECK_MDから完全に消す。

## SETUP_SCRIPT — setup_deck.pyコード

Step 6①でプロジェクトルートに以下のコードを `setup_deck.py` として Write する（既存ファイルがあっても上書き）。

```python
"""deck-from-source の実行ファイル群を /mnt/data にコピーするセットアップスクリプト。
Colab / Jupyter 等の環境で初回セットアップ時に一度実行する。
"""

import shutil
import sys
from pathlib import Path

ROOT = Path(__file__).resolve().parent
SKILL_DIR = ROOT / ".agents" / "skills" / "deck-from-source"
if not (SKILL_DIR / "md_to_json.py").exists():
    SKILL_DIR = ROOT / ".claude" / "skills" / "deck-from-source"
DEST_DIR = Path("/mnt/data")

FILES = [
    "resolve_uploads.py",
    "md_to_json.py",
    "to_pptx.py",
    "template_engine_area.html",
    "templates.json",
    "design.json",
    "logo.png",
    "background.png",
]


def main() -> None:
    missing = [f for f in FILES if not (SKILL_DIR / f).exists()]
    if missing:
        print(f"[エラー] スキルフォルダに以下のファイルが見つかりません: {missing}", file=sys.stderr)
        print(f"  スキルフォルダ: {SKILL_DIR}", file=sys.stderr)
        sys.exit(1)

    DEST_DIR.mkdir(parents=True, exist_ok=True)
    for name in FILES:
        src = SKILL_DIR / name
        dst = DEST_DIR / name
        shutil.copy2(src, dst)
        print(f"  copied: {name}")

    dest = DEST_DIR.resolve()
    print(f"\nセットアップ完了 → {dest}")
    print("\nPPTX生成コマンド:")
    print(f'  python "{dest}/md_to_json.py" deck.md deck.pptx --json deck.json --assets-dir "{dest}"')


if __name__ == "__main__":
    main()
```

**実行後の注意:**
- `/mnt/data` が存在しない環境（ローカルWindowsなど）では失敗する場合があるが、その旨をユーザーに伝えればよい
- Step 6③では、`setup_deck.py` の出力に表示される「PPTX生成コマンド」をそのまま実行すること（パスをハードコードしない）
- Windowsでは `pathlib.Path("/mnt/data").resolve()` は `C:\mnt\data` になるため、出力されたパスを使えば環境差異を吸収できる
- スクリプトファイル群は `.agents/skills/deck-from-source/` または `.claude/skills/deck-from-source/` に置かれている
