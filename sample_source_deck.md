# CLAUDE.mdは固定ルール、MEMORY.mdは学習記録である
## 強制したい前提と、弱い好みを混ぜないことが重要である

| key | value |
|-----|-------|
| layout | plain_2col |
| note | このスライドで伝えたいことは、CLAUDE.mdとMEMORY.mdはどちらも文脈を扱うが、担う役割が根本的に違うという点である。CLAUDE.mdはユーザーが意図的に書く固定ルールであり、AIの行動を明確に制御する。一方、MEMORY.mdはAIが作業から学んだ傾向や好みを保持する補助記憶である。したがって、守らせたいルールはCLAUDE.mdへ置き、弱い好みや最近の作業傾向はMEMORY.mdへ任せる設計が必要である。 |

```card-a
### CLAUDE.md
セッション開始時に読み込まれる固定の行動基準である。
- プロジェクトで必ず守る前提を明文化し、判断のぶれを減らす
- コーディング規約や禁止操作など、逸脱すると危険な内容を置く
- 会話で済む単発希望は入れず、繰り返す指示だけを継続ルールへ昇格させる
```
```card-b
### MEMORY.md / Auto memory
AIが作業から学んだ傾向を補助的に保持する記録領域である。
- よく使う進め方や命名の傾向など、弱い好みを蓄積する
- 最近の失敗や対処ログを残し、次回の探索と確認のコストを下げる
- 強制力は持たせず、判断を助ける参考情報として慎重に扱う領域である
```

# 設計ゴールは厳選・密度・分離の3つである
## ルールを増やすほど良くなるわけではない

| key | value |
|-----|-------|
| layout | list_3card |
| note | このスライドでは、CLAUDE.mdを設計する時の三つのゴールを確認する。第一に、常に守るべき重要ルールだけを厳選することである。第二に、60から300行程度の適切な密度に保ち、重要なルールが埋もれないようにすることである。第三に、プロジェクトやディレクトリごとにスコープを分け、AIが読むべき文脈を絞ることである。この三点により、AIの挙動は場当たり的な推測から、再現性のある判断へ近づく。 --- [nanobanana2 icon prompt] 3つのフラットアイコンを横一列に並べた1枚の画像を生成する。左から順に、アイコン1は不要な紙を選別するフィルター、アイコン2は適度な行数を示す文書メーター、アイコン3は階層化されたフォルダを表す。各アイコンは正方形の枠内に収まるシンプルなフラットデザイン。白背景、色統一、ミニマルスタイル、文字なし。 |

```card-a
### 厳選
常に守るルールだけを残し、判断負荷を下げる設計である。
- 重要でない注意は会話で扱い、固定ルールへ入れず軽さを保つ
- 2回繰り返した指摘だけ、継続ルールとして昇格させる
- ルール過多 → 読み飛ばしと優先順位の混乱を招く
```
```card-b
### 密度
短く具体的な文で、読み返しやすく保つ。
- 60〜300行を目安にし、長文説明より判断条件を優先する
- 抽象的な理想ではなく、実行できるコマンドや判断条件を具体化する
- 何を変え、何を禁止するかを一文単位で明確に示す
```
```card-c
### 分離
必要な場所にだけ効くルールへ分ける。
- 全体方針と局所ルールを混ぜず、読み込み範囲を小さくする
- frontendやinfraなど、作業領域ごとに判断基準を置く
- 関係ない文脈を読ませないことで、AIへのノイズを減らす
```

# 良いルールは実行可能な1文である
## 抽象的な注意を、行動と完了条件へ変換する

| key | value |
|-----|-------|
| layout | table_conclusion |
| note | このスライドでは、CLAUDE.mdに書く文の粒度を説明する。良いルールは、短く、断定調で、実行可能である。不適切な例のように「しっかり行う」と書くと、AIは何をもって完了とするかを推測する必要がある。適切な例のように具体的なコマンドと失敗時の対応まで書けば、作業計画へそのまま組み込める。つまり、CLAUDE.mdでは励ましではなく、行動可能な基準を書くべきである。 |

```table
| 観点 | 不適切な書き方 | 適切な書き方 | 変わる効果 |
| --- | --- | --- | --- |
| テスト | テストはしっかり行う | 変更後は`pnpm test`を実行し、失敗を全件修正する | 完了条件が明確になる |
| 安全性 | 危険な操作に注意する | 破壊的操作はdry-runを用意し、本番前に確認する | 実行前の停止点ができる |
| 文体 | 分かりやすく書く | 1ルール1文で、断定調の実行指示にする | 読み飛ばしを防ぐ |
| スコープ | 必要な時だけ読む | frontend配下では`.claude/rules/frontend.md`を優先する | ノイズが減る |
```
```conclusion
CLAUDE.mdの文は、AIがそのまま作業計画へ入れられる粒度で書くべきである。
```

# スコープは3レイヤーで切る
## 全体・プロジェクト・ディレクトリを分けて効かせる

| key | value |
|-----|-------|
| layout | flow_3step |
| note | このスライドで伝えたいことは、ルールの強さを置き場所で調整することである。グローバル層には個人の共通方針を置き、プロジェクト層には対象技術や危険操作の禁止を置く。ディレクトリ層にはフロントエンドやインフラのような局所ルールを置く。この分離により、AIは現在触っている作業に関係するルールだけを優先し、不要な文脈を背負わずに済む。結果として、読み込み量と判断負荷が同時に下がる。 --- [nanobanana2 icon prompt] 3つのフラットアイコンを横一列に並べた1枚の画像を生成する。左から順に、アイコン1は地球儀と共通ルール、アイコン2はリポジトリ箱と設定ファイル、アイコン3は小さなディレクトリ階層と局所ルールを表す。各アイコンは正方形の枠内に収まるシンプルなフラットデザイン。白背景、色統一、ミニマルスタイル、文字なし。 |

```step-a
### グローバル
全プロジェクト共通の作法である。
- 日本語回答やテスト重視など、横断方針を固定する
- 変えない好みだけを置き、案件固有の前提は避ける
```
```step-b
### プロジェクト
対象リポジトリの基準である。
- 技術スタックと禁止操作を定義し、提案範囲を絞る
- 本番環境 → 上書き不可など、重大事故の条件を書く
```
```step-c
### ディレクトリ
局所ルールを必要箇所へ寄せる。
- UIとインフラで判断基準を分け、関係ない指示を読ませない
- 作業領域に近いルールほど、具体的な手順まで書く
```

# CLAUDE.mdは運用しながら育てる

| key | value |
|-----|-------|
| layout | flow_3step |
| note | このスライドでは、CLAUDE.mdを一度書いて終わりにしない運用を説明する。最初のステップは、会話で同じ注意を二回繰り返したらルールへ昇格させることである。次に、古くなった技術スタックや形骸化した規約を定期的に削る。最後に、小さなタスクで変更後の振る舞いを確認する。CLAUDE.mdは増やす文書ではなく、現場の失敗と学習を反映して軽く保つ運用物である。これにより、ルールは現場の実態に追従し、形骸化を避けられる。 --- [nanobanana2 icon prompt] 3つのフラットアイコンを横一列に並べた1枚の画像を生成する。左から順に、アイコン1は指摘をルールへ昇格する上向き矢印、アイコン2は古い項目を削る整理済み文書、アイコン3は小さなテスト実行と確認マークを表す。各アイコンは正方形の枠内に収まるシンプルなフラットデザイン。白背景、色統一、ミニマルスタイル、文字なし。 |

```step-a
### 昇格する
繰り返し指摘は固定化候補である。

- 摩擦 → 基準化へ変え、次回の迷いを防ぐ
```
```step-b
### 削る
```

