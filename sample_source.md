## 1. CLAUDE.md の正体と設計ゴール

Claude Code には、長期的な文脈を維持するための仕組みが **2つ** あります。

* **CLAUDE.md:** ユーザー自身が記述する「ルール・前提・手順」の指示ファイル。
* **Auto memory / MEMORY.md:** Claude が自動で書き込む「学習したパターンと好み」のメモ。

`CLAUDE.md` は単なる設定ファイルではなく、**「セッションの開始時に必ず読み込まれるルールブック」**です。その目的は、毎回同じ説明を繰り返す手間を省き、AI の挙動をプロジェクトの基準に完全に適合させることにあります。

### 設計の 3 大ゴール
1.  **重要ルールの厳選:** Claude に常に守ってほしいルールのみを記述する（単発の指示は会話で行う）。
2.  **適切な密度:** 60〜300 行程度に抑え、重要なルールが埋もれないようにする。
3.  **スコープの分離:** プロジェクトやディレクトリごとに役割を分け、AI へのノイズを減らす。

## 2. CLAUDE.md と MEMORY.md の役割分担

### 2-1. 役割の違い
両者は「コンテキスト（文脈）」ですが、性質が明確に異なります。

| 項目 | CLAUDE.md | MEMORY.md / Auto memory |
| :--- | :--- | :--- |
| **作成者** | ユーザー（あなた） | Claude（AI） |
| **内容** | 普遍的なルール・前提・手順 | ユーザーの癖・よくやる操作・学んだコツ |
| **読み込み範囲** | 原則全文（サイズ上限まで） | 先頭 200 行 または 25KB まで |
| **スコープ** | プロジェクト / 組織単位 | ワーキングツリー単位 |
| **用途** | 「こう振る舞え」という強制指示 | 「いつもこうしている」という学習結果 |

### 2-2. どちらに何を書くべきか
* **CLAUDE.md に書くもの:**
    * コーディング規約（インデント、命名ルール、禁止事項など）。
    * プロジェクトの目的と主要なアーキテクチャ方針。
    * ビルドやテストに必須のコマンド。
    * 破壊的・危険な操作の禁止指示。
* **MEMORY.md に任せるもの:**
    * 「だいたいいつもこういう設計に寄せる」といった弱い好み。
    * 修正内容から自動で学んだ変数名の傾向などのパターン。
    * 「最近多いエラーの対処法」や「ハマりやすい落とし穴」のログ。

## 3. 長さと情報量のベストプラクティス

### 3-1. 推奨サイズ
* **目安:** 60〜300 行（2〜6 画面程度）。
* Claude が安定して遵守できるルール数は **150〜200 程度** です。
* 「書けるだけ書く」のではなく、**「重要なものだけに絞り込む」**ことが精度向上の鍵となります。

### 3-2. 具体性と粒度
良い `CLAUDE.md` には以下の 3 つの共通点があります。
1.  **1 ルール 1 文:** 短く具体的に書く。
2.  **宣言調:** 「〜してください」ではなく「〜する」と断定する。
3.  **実行可能な指示:** 抽象的なスローガンを避け、行動レベルで指定する。

> **例:**
> * **不適切:** 「テストはしっかり行うこと。」
> * **適切:** 「変更後は `pnpm test` を実行し、失敗したテストをすべて修正する。」

## 4. 推奨される 7 つの章立て

### 4-1. プロジェクト概要 (What)
AI に「何を作っているのか」を理解させます。
* プロダクトの目的（例: 顧客の請求書管理 Web アプリ）。
* 想定ユーザーと重要なユースケース。
* 非機能要件（例: セキュリティ最優先、レスポンス速度重視）。

### 4-2. 技術スタックと禁止事項
提案されるコードの質を揃えます。
* メイン言語・フレームワーク（例: TypeScript + Next.js 14）。
* UI ライブラリの限定（例: Tailwind CSS + Radix UI のみ）。
* 禁止ライブラリ（例: `moment.js` 禁止、`date-fns` を使用）。

### 4-3. ディレクトリ構成とアーキテクチャ
AI がファイルを探すための「地図」です。
* 主要ディレクトリの役割定義（`src/app`, `src/features` など）。
* アーキテクチャ方針（Atomic Design, Clean Architecture など）。
* 「ビジネスロジックは `src/domain` に置き、コンポーネントから直接 fetch しない」といった隔離ルール。

### 4-4. コマンド集
AI が計画に組み込みやすいよう、そのまま実行できる文字列で記述します。
* 開発サーバー: `pnpm dev`
* 単体テスト: `pnpm test`
* 型チェック: `pnpm typecheck`
* lint: `pnpm lint`

### 4-5. コーディング規約 (Style)
自動チェックできない「人間のルール」を明示します。
* インデント: スペース 2 つ（タブ禁止）。
* クォート: JS/TS はシングルクォート。
* 関数: コンポーネントはアロー関数を優先、最大 50 行まで。
* TypeScript: `any` 禁止、`unknown` 優先。

### 4-6. 作業ルール・ワークフロー (Workflow)
Claude Code の「行動」そのものを制御します。
* マルチファイル変更時は、まず調査してから計画を提示する。
* 変更後は必ず関連テスト → lint → typecheck を実行する。
* 「コンパクション（要約）時、変更ファイル一覧とテスト結果は必ず保持する」といった記憶の保持指示。

### 4-7. プロジェクト固有の落とし穴 (Gotchas)
推測が難しい、現場特有の制約を書きます。
* 本番環境の `DATABASE_URL` は絶対に上書きしない。
* ログには個人情報を絶対に含めない（マスキング必須）。
* 実行環境の前提（WSL 前提、PowerShell 対応など）。

## 5. rules とスコープ設計

大規模リポジトリでは `.claude/rules/` ディレクトリを活用し、ディレクトリごとにルールを適用します。

* **ルート CLAUDE.md:** プロジェクト全体の共通ルール。
* **frontend.md:** UI 関連のディレクトリのみに適用。
* **infra.md:** スクリプトやインフラ関連のみに適用。

これにより、AI が今触っているファイルに関係のあるルールだけを優先的に読み込み、認知負荷を下げることができます。

## 6. 用途別テンプレートのポイント

### 6-1. Web アプリ向け
* CSS フレームワークの指定。
* 状態管理ライブラリ（Zustand, TanStack Query 等）の使用方針。
* API 呼び出しの配置ルール。

### 6-2. データ分析 / 機械学習向け
* 使用ライブラリの固定（pandas, scikit-learn 等）。
* データの保存場所（`data/raw`, `data/processed`）の厳守。
* 再現性ポリシー（ランダムシードの固定など）。

### 6-3. スクリプト / 自動化向け
* **対象 OS / シェルの明示:** Windows / PowerShell / WSL / Linux。
* **安全策:** 「破壊的操作は `dry-run` を用意する」「本番実行は `-WhatIf` を挟む」。
* **エラー処理:** すべてのスクリプトは `logs/` にローテーションログを出す。

## 7. 運用テクニック：CLAUDE.md の育て方

1.  **「2 回言ったら CLAUDE.md 行き」:** 会話で同じ注意を 2 回繰り返したら、即座にルールへ昇格させる。
2.  **定期的なリファクタリング:** 古くなった技術スタックや、形骸化したルールを削る。
3.  **変更のテスト:** ルールを書き換えた後は、小さなタスクを投げて AI の振る舞いが期待通りか確認する。

## 8. Takuya さん向けの設計指針

あなたの環境では、以下の **3 レイヤー構造** が最も効果的です。

1.  **グローバル:** 自身の全プロジェクト共通の「性格・好み」（例: テスト必須、日本語回答）。
2.  **プロジェクト単位:** 対象技術やディレクトリ構成、危険な操作の禁止事項。
3.  **ディレクトリ単位 (.claude/rules):** フロントエンド UI と、PowerShell/Python インフラ操作のルールを分離。
