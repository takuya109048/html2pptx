# Source-to-Deck: 原文からスライドデッキMD生成スキル

原文テキストを受け取り、`sample_deck.md` と同じ記述ルールに従ったスライドデッキ用Markdownを生成するスキル。

---

## トリガー条件

- 「原文からスライドを作って」「記事をデッキにして」「〇〇をプレゼン資料に」などの依頼
- ユーザーが本文テキスト・URL・ファイルを提示し、スライド化を求めている場合

---

## Step 0: 原文の受け取り

ユーザーが原文をまだ提供していない場合、以下のいずれかを確認する:

1. **テキスト貼り付け**: 原文を直接チャットに貼り付けてもらう
2. **URL指定**: URLを教えてもらい `WebFetch` で取得する
3. **ファイル指定**: ファイルパスを教えてもらい `Read` で読み込む

原文を受け取ったら、以下を確認する:
- 想定する **発表時間・スライド枚数** （未指定なら枚数を提案する）
- **発表者名・所属・発表日** （coverスライドに使用）
- **対象オーディエンス** （技術者向け・経営層向けなど）

---

## Step 1: 原文の分析

原文を読み込み、以下を抽出する:

1. **主題・タイトル**: デッキ全体のタイトルになるもの
2. **論点の階層**: 大項目（スライドタイトル候補）と小項目（箇条書き候補）
3. **データ・比較情報**: 表（table）に適した数値・比較データ
4. **プロセス・手順**: フロー（flow_3step / flow_4step）に適した段階的情報
5. **分類・類型**: カード（list_3card / diffuse_3card / converge_3card / bg_3card）に適した並列概念
6. **結論・示唆**: 結論スライドや converge / table_conclusion に使えるまとめ

分析結果をもとに **スライド構成案**（スライド番号・タイトル・使用レイアウト）を箇条書きで提示し、ユーザーの確認を得る。

---

## Step 2: レイアウト選択ガイド

各スライドのコンテンツ特性に合わせて以下から選ぶ:

| layout | 使う場面 |
|---|---|
| `cover` | 表紙（必ず1枚目） |
| `plain_1col` | 1テーマを本文テキスト＋箇条書きで説明 |
| `plain_2col` | 2つの観点を左右に並べて比較・説明 |
| `list_3card` | 並列する3つの概念・施策・特徴 |
| `flow_3step` | 3段階のプロセス・ステップ・フロー |
| `flow_4step` | 4段階のプロセス・ステップ・フロー |
| `table` | 数値データや比較を表で示す |
| `table_conclusion` | 表＋結論を1スライドにまとめる |
| `diffuse_3card` | 1つの共通方針から3つの施策へ展開 |
| `converge_3card` | 3つの根拠・観点から1つの結論へ収束 |
| `bg_3card` | 背景文脈＋横並び3カードで強調 |
| `plain_image_row` | テキスト＋縦配置の画像プレースホルダー |
| `plain_image_col` | テキスト＋横配置の画像プレースホルダー |

---

## Step 3: Markdown生成ルール

### 3-1. スライドの基本構造

```
# スライドタイトル

| key | value |
|-----|-------|
| layout | <レイアウト名> |
| note | 発表者ノート（原文の該当箇所を要約したもの） |

## <タグ名>
...コンテンツ...

---
```

### 3-2. coverスライド

```
# デッキタイトル

| key | value |
|-----|-------|
| layout | cover |
| affiliation | 所属・組織名 |
| presenter | 発表者名 |
| date | YYYY-MM-DD |
| note | 発表の背景・目的を1〜2文で。 |

---
```

### 3-3. タグのルール

- `card-a` / `card-b` / `card-c`: list_3card, plain_2col, diffuse_3card, converge_3card, bg_3card で使用
- `step-a` / `step-b` / `step-c` / `step-d`: flow_3step / flow_4step で使用
- `section`: diffuse_3card / bg_3card の共通見出し
- `conclusion`: table_conclusion / converge_3card の結論
- `table`: table / table_conclusion の表（Markdownテーブル必須）

各カード・ステップには `### 見出し` を付け、本文は箇条書き（`-`）または短文で記述する。

### 3-4. Markdownインライン書式

原文のニュアンスに応じて以下を使う:

| 書式 | 用途 |
|---|---|
| `**太字**` | 重要キーワード・数値・強調 |
| `*斜体*` | 補足・定義・術語 |
| `\` | 固有名詞・型番・コマンド名 |
| `~~取り消し~~` | 旧情報・廃止・否定的要素 |

### 3-5. noteフィールド

- 各スライドの `note` には、**発表者が口頭で補足すべき内容**を原文から抽出して記載する
- 原文の具体的な数値・引用・背景説明を優先的に `note` に入れる
- 1〜3文程度に収める

### 3-6. スライド枚数の目安

| 発表時間 | 推奨枚数 |
|---|---|
| 5分 | 5〜7枚（cover含む） |
| 10分 | 8〜12枚 |
| 15〜20分 | 12〜18枚 |
| 30分以上 | 18〜25枚 |

### 3-7. タイトル・メッセージ行は必ず記述する

- `# タイトル` と `## メッセージ`（サブタイトル）は**内容がある場合は必ず記述する**
- タイトルやメッセージを省略すると、レンダラーがプレースホルダーテキストに置き換えてしまう
- cover レイアウトではタイトル行（`#`）は必須。`## メッセージ` は不要なら省略可
- それ以外のレイアウトでは `# タイトル` は必須。`## メッセージ` は情報があれば記載する

---

## Step 4: 出力

以下の形式で出力する:

1. **スライドデッキMD全文** をコードブロック（` ```md ` ）に入れて出力する
2. 出力後、ユーザーに以下を確認する:
   - 修正・差し替えたいスライドがあれば指示してほしい
   - PPTXに変換する場合は `python md_to_json.py <ファイル名>.md` で実行できる旨を案内する

## 注意事項

- 原文にない情報を**作り上げない**（ハルシネーション禁止）
- 原文の重要度・論旨に忠実にスライド化する
- 1スライドに詰め込みすぎない（1スライド1メッセージが原則）
- 画像が必要な場面では `plain_image_row` / `plain_image_col` を使い `image_label_1` でプレースホルダーテキストを指定する
